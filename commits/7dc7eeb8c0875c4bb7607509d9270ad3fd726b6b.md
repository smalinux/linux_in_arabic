## Explain It Like I'm 5

Imagine your toy car runs on a special engine that gets faster and slower depending on how you spin a dial. Now imagine that while the car is moving, you **cannot** adjust the dial or the whole car crashes and breaks.

To fix this: before you touch the dial, you **switch the car to run on a small backup battery** (very slow, but safe and always works). Then you adjust the dial freely. Once the dial is set, you switch the car back to the real engine.

This commit does exactly that for a real computer chip (the **Amlogic Meson8b** SoC). The CPU runs off a big clock generator called the **SYS PLL**. Changing the CPU speed requires reconfiguring that PLL — but while the CPU depends on it, touching the PLL freezes the whole computer. The fix: before changing anything, switch the CPU to use a tiny safe crystal clock (XTAL), change the PLL, then switch back.

---

## Background: The Hardware

**Amlogic Meson8b** is a System-on-Chip (SoC) used in ARM-based devices like Android TV boxes and set-top boxes. It has a driver in the Linux kernel clock subsystem (`drivers/clk/meson/`).

### Key Clock Sources

| Clock | Description | Frequency |
|-------|-------------|-----------|
| **XTAL** | Crystal oscillator — always on, perfectly stable | 24 MHz (fixed) |
| **SYS PLL** | Phase-Locked Loop — multiplies XTAL to make high-speed clocks | ~600 MHz – 1.5 GHz (variable) |

---

## The CPU Clock Tree

The CPU clock is not taken directly from the PLL — it goes through a chain of muxes and dividers:

```
XTAL (24 MHz) ─────────────────────────────────┐
                                                ▼
XTAL → sys_pll_dco → sys_pll → cpu_in_sel ─→ cpu_div2        ┐
                                            ├─→ cpu_div3        │→ cpu_scale_out_sel → cpu_clk
                                            └─→ cpu_scale_div  ┘          │
                                                                           │
XTAL (24 MHz) ─────────────────────────────────────────────────────────── ┘
(direct bypass, the "safe path")
```

The final **cpu_clk mux** has two parents:
1. `xtal` — direct 24 MHz (the safe bypass)
2. `cpu_scale_out_sel` — the high-speed path through the PLL chain

This is defined in the code as:

```c
static struct clk_regmap meson8b_cpu_clk = {
    .hw.init = &(struct clk_init_data){
        .name = "cpu_clk",
        .parent_names = (const char *[]){ "xtal", "cpu_scale_out_sel" },
        .num_parents = 2,
        .flags = (CLK_SET_RATE_PARENT | CLK_SET_RATE_NO_REPARENT | CLK_IS_CRITICAL),
    },
};
```

Notice `CLK_SET_RATE_NO_REPARENT` — it tells the kernel: **do not automatically reparent this clock when rates change**. That flag is important because the kernel would normally try to auto-switch parents, but here we need manual control.

## The Problem

When Linux wants to change the CPU frequency (e.g. cpufreq scaling from 600 MHz to 1 GHz), it must reprogram the **SYS PLL** registers. Reprogramming a PLL involves:

1. Changing the N/M multiplier values in hardware registers
2. Waiting for the PLL to lock onto the new frequency
3. The PLL output is **glitchy and unstable** during steps 1–2

While this is happening, the CPU is still being clocked **through** the SYS PLL. The CPU receives a garbage clock signal and the **whole system hard-locks** (complete freeze, no recovery except power cycle).

---

## The Fix: A Clock Change Notifier

The solution is a **Linux clock notifier** — a callback function that the kernel clock framework calls **before** and **after** any rate change propagates through a clock node.

### What clock is watched?

The notifier is registered on **`cpu_scale_out_sel`** — the direct parent of `cpu_clk` in the high-speed path. Whenever anything in the PLL chain changes rate, the notification propagates up to `cpu_scale_out_sel` and triggers the callback.

### New struct: `meson8b_nb_data`

```c
struct meson8b_nb_data {
    struct notifier_block nb;           // standard kernel notifier hook
    struct clk_hw_onecell_data *onecell_data; // pointer to all clock handles
};
```

This wraps the standard `notifier_block` with a pointer to the clock table so the callback can look up `CLKID_XTAL`, `CLKID_CPUCLK`, and `CLKID_CPU_SCALE_OUT_SEL` by index.

### The callback: `meson8b_cpu_clk_notifier_cb`

```c
static int meson8b_cpu_clk_notifier_cb(struct notifier_block *nb,
                                       unsigned long event, void *data)
{
    // get our wrapper struct from the embedded notifier_block
    struct meson8b_nb_data *nb_data =
        container_of(nb, struct meson8b_nb_data, nb);

    switch (event) {
    case PRE_RATE_CHANGE:
        // BEFORE the PLL changes: switch CPU to safe XTAL
        parent_clk_hw = hws[CLKID_XTAL];
        break;

    case POST_RATE_CHANGE:
        // AFTER the PLL has settled: switch CPU back to normal path
        parent_clk_hw = hws[CLKID_CPU_SCALE_OUT_SEL];
        break;

    default:
        return NOTIFY_DONE; // ignore ABORT_RATE_CHANGE etc.
    }

    cpu_clk_hw = hws[CLKID_CPUCLK];
    cpu_clk    = __clk_lookup(clk_hw_get_name(cpu_clk_hw));
    parent_clk = __clk_lookup(clk_hw_get_name(parent_clk_hw));

    ret = clk_set_parent(cpu_clk, parent_clk); // atomically reparent
    if (ret)
        return notifier_from_errno(ret);

    udelay(100); // wait 100 µs for the new clock to stabilize

    return NOTIFY_OK;
}
```

### Timeline of a CPU frequency change

```
cpufreq driver requests 1 GHz
        │
        ▼
clk_set_rate(sys_pll) called
        │
        ├─ PRE_RATE_CHANGE event fires
        │       └─ notifier callback: cpu_clk → XTAL (24 MHz)
        │          udelay(100 µs)          CPU now safe at 24 MHz
        │
        ├─ PLL registers reprogrammed (glitchy period, harmless now)
        │
        ├─ PLL locks at 1 GHz
        │
        └─ POST_RATE_CHANGE event fires
                └─ notifier callback: cpu_clk → cpu_scale_out_sel (1 GHz path)
                   udelay(100 µs)          CPU now running at 1 GHz
```

## Registration in `meson8b_clkc_init`

At boot, the notifier is wired up in the init function:

```c
notifier_clk_name = clk_hw_get_name(&meson8b_cpu_scale_out_sel.hw);
notifier_clk = __clk_lookup(notifier_clk_name);
ret = clk_notifier_register(notifier_clk, &meson8b_cpu_nb_data.nb);
```

This says: **whenever `cpu_scale_out_sel` is about to change rate, call our callback**.

---

## The `container_of` Pattern

```c
struct meson8b_nb_data *nb_data =
    container_of(nb, struct meson8b_nb_data, nb);
```

The kernel passes a bare `struct notifier_block *` to the callback. But our struct `meson8b_nb_data` **embeds** a `notifier_block` as its first-ish member. `container_of` is a C macro that calculates the address of the outer struct from the address of an inner member, using compile-time offset arithmetic. This is a very common Linux kernel pattern for OOP-like polymorphism in C.

---

## The `udelay(100)` — Why 100 Microseconds?

After calling `clk_set_parent()`, the hardware mux in the SoC switches to the new parent. The 100 µs delay is a conservative safety margin to let:
- The new clock path propagate through the digital logic
- Any residual glitches settle
- The CPU pipeline flush and restart cleanly on the new clock domain

This is a hardware-specific empirical value — no formula, just experience with this chip.

## The FIXME Comment

```c
/*
 * FIXME we shouldn't program the muxes in notifier handlers. The
 * tricky programming sequence will be handled by the forthcoming
 * coordinated clock rates mechanism once that feature is released.
 */
```

The author is being honest: this is a **workaround**, not the ideal solution. The Linux clock framework was being extended with a "coordinated clock rates" mechanism that would handle these complex multi-clock switching sequences properly. Since that feature wasn't ready yet, the notifier hack was the pragmatic fix. This is a common pattern in kernel development: ship a working fix now, replace it cleanly later.

---

## What Would Have Happened Without This Fix?

1. User installs a CPU frequency scaling governor (e.g. `schedutil`)
2. Kernel tries to boost CPU from 600 MHz to 1 GHz under load
3. `clk_set_rate()` reprograms SYS PLL registers
4. PLL output goes glitchy for ~10–100 µs
5. CPU receives a corrupted clock
6. CPU misexecutes instructions, pipeline corrupts
7. **Complete system freeze** — no panic, no log, just silence
8. Only recovery: power cycle the device
