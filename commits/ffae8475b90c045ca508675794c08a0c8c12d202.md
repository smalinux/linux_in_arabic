### تخيل معايا...

تخيل إنك بتشغّل محرك سيارة بيتغذى من مولّد كهربي، وأنت عايز تغيّر سرعة المولّد (تزوده أو تخففه) من غير ما المحرك يتوقف أو يتعطل.

المشكلة: لو غيّرت سرعة المولّد وهو شاغل المحرك مباشرة، هيحصل "لحظة" المحرك مش بياخد تيار صح — ده ممكن يخليه يتعطل أو يشتغل بشكل غلط.

الحل الصح: قبل ما تغيّر المولّد، توصّل المحرك بـ**مصدر احتياطي ثابت وآمن** (زي بطارية بـ 24V ثابتة). بعدين تغيّر سرعة المولّد براحتك. لما المولّد يستقر على الترددات الجديدة، ترجع توصّل المحرك بيه.

ده **بالظبط** اللي بيحصل في الـ CPU clock switching على شريحة Amlogic G12A/G12B:
- **الـ CPU** = المحرك
- **الـ sys_pll** = المولّد الرئيسي اللي بيشغّل الـ CPU على ترددات عالية (GHz)
- **الـ xtal** (24 MHz) = البطارية الاحتياطية الثابتة
- **الـ notifiers** = الـ "ريموت" اللي بيقول "وصّل البطارية الاحتياطية" قبل التغيير، وبعدين "رجّع المولّد" لما خلص

قبل الـ commit ده، مفيش أي "ريموت" — الكيرنل كان بيغيّر تردد الـ PLL مباشرة وهو شاغل الـ CPU، وده ممكن يسبب تعليق أو crash.

### الصورة الكبيرة

**الـ subsystem:** Linux Clock Framework (CCF) — **الـ** `drivers/clk/meson/`

**الـ** CCF (Common Clock Framework) هو المسؤول عن إدارة كل الـ clocks جوه الكيرنل. لما الـ cpufreq driver يطلب تغيير تردد الـ CPU (مثلاً من 1.8 GHz لـ 1.2 GHz)، بيطلب من الـ CCF يعمل كده.

**قبل الـ commit:**

- كل الـ clocks في شجرة الـ CPU على G12A/G12B كانت متعرّفة بـ `_ro_ops` (read-only operations) — يعني الكيرنل ما كانش يقدر يغيّرها خالص.
- الـ sys_pll_dco كانت بتستخدم قائمة `init_regs` صنع-ونسى — مش بتدعم تغيير التردد ديناميكياً.
- مفيش أي **notifiers** تتحكم في تسلسل التغيير.

**النتيجة:** مكانش ممكن عمل CPU frequency scaling (DVFS) أصلاً على G12A/G12B بشكل صحيح.

**بعد الـ commit:**

1. كل الـ clocks في شجرة الـ CPU اتحوّلت من `_ro_ops` لـ `_ops` — يعني قابلة للتعديل.
2. اتضاف نظام **notifiers** من 3 طبقات يضمن إن التغيير يحصل بدون glitches:
   - **Notifier 1** على `cpu_clk_postmux0`: لما الـ CCF عايز يغيّر الـ sub-GHz dynamic path، ينقل الـ CPU لـ "parking clock" (xtal 24 MHz) مؤقتاً.
   - **Notifier 2** على `sys_pll`/`sys1_pll`: لما الـ CCF عايز يغيّر الـ PLL الرئيسي، ينقل الـ CPU للـ `cpu_clk_dyn` path مؤقتاً.
   - **Notifier 3** على `cpu_clk` و`cpu_clk_dyn`: يضيف `udelay(100)` قبل وبعد كل تغيير عشان يدي الـ notifiers التانية وقت كافي تشتغل.
3. الـ `cpu_clk_premux1` sub-tree اتعيّنت كـ "parking clock" ثابتة بـ `CLK_SET_RATE_NO_REPARENT`.
4. الـ `sys_pll_dco` اتحوّل لـ `CLK_IS_CRITICAL` — مش هيتقفل أبداً لأنه بيغذي الـ CPU.

**ليه ده مهم لـ end users؟**

بدون الـ commit ده، الـ cpufreq (اللي بيتحكم في استهلاك الطاقة وأداء الجهاز) مش هيشتغل على أي جهاز بيستخدم G12A/G12B. الجهاز هيشتغل على تردد ثابت طول الوقت — يعني إما بطاريته بتخلص بسرعة، أو بيشتغل ببطء. مع الـ commit ده، cpufreq بيشتغل صح وبدون crash.

## Phase 2: المشكلة — ليه الكود القديم كان بيتعطل؟

### الكود قبل التعديل

قبل الـ commit ده، كل clocks في شجرة الـ CPU — من الـ `sys_pll_dco` لحد الـ `cpu_clk` — كانت معرّفة بـ **read-only ops**، يعني الـ CCF (Common Clock Framework) ممنوع يغيّر فيها أي حاجة:

```c
/* sys_pll_dco: PLL الرئيسي اللي بيشغّل الـ CPU */
static struct clk_regmap g12a_sys_pll_dco = {
    .data = &(struct meson_clk_pll_data){
        /* ... hardware fields ... */
        .init_regs = g12a_sys_init_regs,   // init registers hardcoded
        .init_count = ARRAY_SIZE(g12a_sys_init_regs),
    },
    .hw.init = &(struct clk_init_data){
        .name = "sys_pll_dco",
        .ops = &meson_clk_pll_ro_ops,      // READ-ONLY: no set_rate!
        /* no CLK_IS_CRITICAL flag */
    },
};

/* sys_pll: divider فوق الـ DCO */
static struct clk_regmap g12a_sys_pll = {
    .hw.init = &(struct clk_init_data){
        .name = "sys_pll",
        .ops = &clk_regmap_divider_ro_ops, // READ-ONLY: no set_rate!
        /* no CLK_SET_RATE_PARENT */
    },
};

/* cpu_clk_premux0: mux بيختار المصدر للـ dyn path */
static struct clk_regmap g12a_cpu_clk_premux0 = {
    .hw.init = &(struct clk_init_data){
        .name = "cpu_clk_dyn0_sel",
        .ops = &clk_regmap_mux_ro_ops,     // READ-ONLY: no set_parent!
        /* no CLK_SET_RATE_NO_REPARENT */
    },
};

/* cpu_clk_mux0_div: divider بيقسم الـ clock */
static struct clk_regmap g12a_cpu_clk_mux0_div = {
    .data = &(struct clk_regmap_div_data){ // simple divider فقط
        .offset = HHI_SYS_CPU_CLK_CNTL0,
        .shift = 4,
        .width = 6,
    },
    .hw.init = &(struct clk_init_data){
        .name = "cpu_clk_dyn0_div",
        .ops = &clk_regmap_divider_ro_ops, // READ-ONLY
    },
};

/* cpu_clk_dyn: مux بيختار بين postmux0 وpostmux1 */
static struct clk_regmap g12a_cpu_clk_dyn = {
    .hw.init = &(struct clk_init_data){
        .name = "cpu_clk_dyn",
        .ops = &clk_regmap_mux_ro_ops,     // READ-ONLY: CCF ميقدرش يغيّر
    },
};

/* cpu_clk: الـ mux الأخير — sys_pll أو cpu_clk_dyn */
static struct clk_regmap g12a_cpu_clk = {
    .hw.init = &(struct clk_init_data){
        .name = "cpu_clk",
        .ops = &clk_regmap_mux_ro_ops,     // READ-ONLY: ممنوع التغيير!
    },
};
```

**الـ `meson_clk_pll_ro_ops`** و**`clk_regmap_mux_ro_ops`** كلاهما بيعملوا `recalc_rate`/`get_parent` بس — من غير `set_rate` أو `set_parent`:

```c
// clk-pll.c
const struct clk_ops meson_clk_pll_ro_ops = {
    .recalc_rate = meson_clk_pll_recalc_rate,
    .is_enabled  = meson_clk_pll_is_enabled,
    // NO set_rate, NO round_rate, NO init
};

// clk-regmap.c
const struct clk_ops clk_regmap_mux_ro_ops = {
    .init       = clk_regmap_init,
    .get_parent = clk_regmap_mux_get_parent,
    // NO set_parent, NO determine_rate
};
```

### الـ Architecture قبل الـ fix

شجرة الـ clock كانت كالآتي — كل العقد بـ `[RO]` يعني read-only، ميقدرش الـ CCF يبعت لها `set_rate`:

```
xtal (24 MHz)
│
├── fixed_pll_dco [RO]
│   └── fixed_pll [RO]
│       ├── fclk_div2
│       └── fclk_div3
│
└── sys_pll_dco [RO] ◄── PLL الرئيسي للـ CPU
    └── sys_pll [RO]
        │
        └── cpu_clk [RO] ◄── الـ mux الأخير (shift=11)
            ├── [parent0] cpu_clk_dyn [RO] ◄── الـ dyn mux (shift=10)
            │   ├── [parent0] cpu_clk_postmux0 [RO]
            │   │   ├── [parent0] cpu_clk_premux0 [RO] (xtal/fclk_div2/fclk_div3)
            │   │   └── [parent1] cpu_clk_mux0_div [RO] ← simple divider
            │   │                  └── cpu_clk_premux0 [RO]
            │   └── [parent1] cpu_clk_postmux1 [RO]
            │       ├── [parent0] cpu_clk_premux1 [RO]
            │       └── [parent1] cpu_clk_mux1_div [RO]
            │
            └── [parent1] sys_pll [RO]
                               ▲
                     ┌─────────┘
           sys_pll_dco [RO]  ←── POINT OF FAILURE
           (PLL بيعمل glitch أثناء تغيير التردد)
```

النقطة الخطيرة: لو الـ cpufreq driver أو أي driver حاول يطلب تغيير تردد الـ CPU، الـ CCF كان هيرفض لأن مفيش `set_rate` على أي node في الشجرة دي.

### ليه كان بيحصل crash / lockup / bug؟

**المشكلة في الواقع مشكلتين مترابطتين:**

#### المشكلة الأولى — الـ CCF عاجز عن تغيير التردد أصلاً

1. الـ cpufreq driver (أو أي consumer) بيبعت `clk_set_rate(cpu_clk, new_freq)`.
2. الـ CCF بيمشي في شجرة الـ clock من فوق لتحت يدوّر على node عندها `set_rate`.
3. كل node في الشجرة عندها `ro_ops` — مفيش `set_rate` في أي مكان.
4. نتيجة: الطلب بيفشل، التردد ميتغيرش، الـ cpufreq ميشتغلش.

#### المشكلة الثانية — الـ PLL glitch لو اتجاوز الحماية

حتى لو فرضنا إن حد غيّر التردد يدوياً (via debugfs أو bootloader setup)، الـ sys_pll_dco لما بيغيّر قيم الـ M/N registers بتاعته بيعمل الخطوات دي:

```
1. meson_clk_pll_disable(hw)   → بيوقف الـ PLL → cpu_clk بيوقف مؤقتاً!
2. meson_parm_write(clk->map, &pll->n, n)  → بيكتب الـ N الجديد
3. meson_parm_write(clk->map, &pll->m, m)  → بيكتب الـ M الجديد
4. meson_clk_pll_enable(hw)    → بيشغّل الـ PLL تاني
5. meson_clk_pll_wait_lock(hw) → بيستنى lock (بياخد وقت)
```

**في الخطوة 1**: لو الـ `cpu_clk` لسه متوصل بالـ `sys_pll` (الـ mux عند bit 11 = 1)، لحظة إيقاف الـ PLL = توقف تام للـ CPU clock = **system freeze** أو CPU reset.

**لأن** `cpu_clk_dyn` و`cpu_clk` كلهم بـ `mux_ro_ops`، مفيش آلية تنقل الـ CPU لمصدر آمن (زي `xtal` أو `fclk_div3`) قبل ما الـ PLL يتوقف.

#### المشكلة الثالثة — الـ dynamic divider بدون atomic update

الـ `cpu_clk_mux0_div` كان بيستخدم `clk_regmap_div_data` (simple divider)، يعني لما بتكتب قيمة الـ divider الجديدة في الـ register، التغيير بيحصل فوراً في نفس الـ clock cycle — **glitch مباشر** على الـ CPU clock.

الـ G12A hardware عنده register field خاص (bit 26 في `HHI_SYS_CPU_CLK_CNTL0`) اسمه `dyn_enable` بيخلّي الـ hardware ياخد القيمة الجديدة بشكل atomic من غير قطع. الكود القديم كان بيتجاهل الـ field ده تماماً.

#### تسلسل الأحداث الكامل للـ crash:

```
[User/cpufreq]  clk_set_rate(cpu_clk, 1.5GHz)
      │
      ▼
[CCF]  يبحث عن set_rate في الشجرة
       cpu_clk       → mux_ro_ops    → ✗ مفيش set_rate
       cpu_clk_dyn   → mux_ro_ops    → ✗ مفيش set_rate
       sys_pll       → divider_ro_ops → ✗ مفيش set_rate
       sys_pll_dco   → pll_ro_ops    → ✗ مفيش set_rate
      │
      ▼
[CCF]  يرفض الطلب → -ENOTSUPP
      │
      ▼
[Result] التردد ثابت زي ما الـ bootloader سبّه
         الـ cpufreq governor عاجز يغيّر أي حاجة
         لو الجهاز اشتغل بتردد عالي في الحرارة → overheat + crash
```

**لو جرت محاولة تغيير مباشرة على الـ PLL** (سيناريو أخطر):

```
[sys_pll_dco set_rate]
  Step 1: meson_clk_pll_disable()
          → sys_pll_dco يوقف VCO
          → sys_pll تبقى undefined/glitched
          → cpu_clk_dyn مازال متوصل بـ sys_pll
          → CPU يشتغل بتردد غير محدد لحظياً
  Step 2-3: كتابة M/N الجدد
          → during this time: cpu_clk = GARBAGE FREQUENCY
          → إمكانية: CPU يتعطل، pipeline corruption
  Step 4: meson_clk_pll_enable()
          → PLL بيحاول يعمل lock
          → lock بياخد microseconds
          → CPU ماشي بتردد غلط طول الوقت ده
  Step 5: [إذا فشل lock] → recursive call → infinite loop!
```

### إيه اللي كان بيحصل للمستخدم؟

| السيناريو | ما يحدث للمستخدم |
|-----------|-----------------|
| **cpufreq عاجز** | الـ CPU بيشتغل على تردد واحد ثابت طول الوقت (اللي سبّه الـ bootloader). الـ scaling مش شغّال خالص. |
| **CPU تقيل + حرارة عالية** | مفيش power saving → استهلاك بطارية أعلى، حرارة ترتفع، خطر thermal throttling أو shutdown. |
| **محاولة تغيير PLL مباشرة** | system freeze أو kernel panic لو الـ CPU اشتغل بدون clock لمدة كافية. |
| **الـ glitch على الـ divider** | instruction corruption لو الـ pipeline كان نشطاً لحظة تغيير الـ divider. |

**هل كان deterministic؟** — عدم الاستجابة لطلبات الـ cpufreq كان **دائم ومضمون** (لأن الـ ops مش موجودة). أما الـ crash من الـ PLL glitch فكان **نادر وغير متوقع** يحصل بس لو حد جرّب يغيّر التردد من خارج الـ CCF أو في حالات edge cases.

**هل كان في workaround؟** — الـ workaround الوحيد الموجود كان إن الـ bootloader يسبّ التردد على قيمة ثابتة والـ Linux يسيبه زي ما هو. لكن ده معناه إن G12A و G12B هيشتغلوا على تردد واحد طول الوقت بدون أي dynamic frequency scaling — مش مقبول لأجهزة الـ production.
## Phase 3: التحليل التقني — إزاي الـ fix اشتغل؟

### الـ Structs والـ Types الجديدة

#### 1. `struct pll_mult_range` — تحديد نطاق الـ multiplier

```c
static const struct pll_mult_range g12a_sys_pll_mult_range = {
    .min = 128,
    .max = 250,
};
```

**الـ** `struct pll_mult_range` موجودة في `clk-pll.h`:

```c
struct pll_mult_range {
    unsigned int    min;
    unsigned int    max;
};
```

دي struct بسيطة بتحدد الحدود المسموح بيها للـ multiplier بتاع الـ PLL (يعني القيمة اللي بتتضرب في تردد الـ xtal عشان نوصل للتردد الكامل للـ PLL). الـ sys_pll بتاع G12A بيقبل multiplier من 128 لـ 250، ده بيتحول لـ تردد من `24MHz × 128 = 3072MHz` كحد أدنى إلى `24MHz × 250 = 6000MHz` كحد أقصى — وطبعاً بعد قسمة الـ divider.

الفرق عن الكود القديم: قبل الـ commit ده، الـ sys_pll كانت بتشتغل بـ `meson_clk_pll_ro_ops` (read-only) وكانت عندها `init_regs` بتحط قيم hard-coded في registers الـ PLL. الـ commit استبدل ده كله بـ `range` + `meson_clk_pll_ops` عشان الـ CCF (Common Clock Framework) يقدر يتحكم في التردد فعلياً.

#### 2. `struct meson_clk_cpu_dyndiv_data` — بديل الـ divider العادي

```c
struct meson_clk_cpu_dyndiv_data {
    struct parm div;   /* register field للـ divider value */
    struct parm dyn;   /* register field لـ dynamic enable bit */
};
```

**الـ** `struct parm` هي:

```c
struct parm {
    u16  reg_off;   /* offset الـ register من base address */
    u8   shift;     /* بداية الـ bit field */
    u8   width;     /* عرض الـ bit field بالـ bits */
};
```

الـ data ده بيوصف **fieldين** في نفس الـ register:

- **`div`**: الـ field اللي فيه قيمة القسمة (مثلاً bits 4:9 من `HHI_SYS_CPU_CLK_CNTL0`)
- **`dyn`**: الـ "dynamic enable" bit (bit 26) — ده البت السحري اللي بيخلي الـ hardware يعمل الـ clock transition بدون glitch

الاستخدام في الكود:

```c
static struct clk_regmap g12a_cpu_clk_mux0_div = {
    .data = &(struct meson_clk_cpu_dyndiv_data){
        .div = {
            .reg_off = HHI_SYS_CPU_CLK_CNTL0,
            .shift = 4,
            .width = 6,
        },
        .dyn = {
            .reg_off = HHI_SYS_CPU_CLK_CNTL0,
            .shift = 26,
            .width = 1,
        },
    },
    ...
    .ops = &meson_clk_cpu_dyndiv_ops,
};
```

ليه استبدلنا `clk_regmap_div_data` بـ `meson_clk_cpu_dyndiv_data`؟ لأن الـ divider العادي بيكتب قيمة الـ divider مباشرة على طول، وده بيعمل clock glitch لحظة الكتابة. الـ `meson_clk_cpu_dyndiv_ops` بتكتب الـ `dyn` bit الأول **قبل** ما تغير الـ divider، والـ hardware بيفهم إن في transition قادم فيستعد ليه.

### الـ Functions الجديدة — سطر بسطر

#### 1. `g12a_cpu_clk_mux_notifier_cb` — الـ notifier البسيط

```c
static int g12a_cpu_clk_mux_notifier_cb(struct notifier_block *nb,
                    unsigned long event, void *data)
{
    if (event == POST_RATE_CHANGE || event == PRE_RATE_CHANGE) {
        /* Wait for clock propagation before/after changing the mux */
        udelay(100);
        return NOTIFY_OK;
    }

    return NOTIFY_DONE;
}

static struct notifier_block g12a_cpu_clk_mux_nb = {
    .notifier_call = g12a_cpu_clk_mux_notifier_cb,
};
```

**إيه اللي بيعمله؟**

ده أبسط notifier في الـ commit. بيتسجل على:
- `cpu_clk` (G12A و G12B)
- `cpub_clk` (G12B)
- `cpu_clk_dyn` (G12A و G12B)
- `cpub_clk_dyn` (G12B)

**وظيفته الوحيدة**: `udelay(100)` — ينتظر 100 microsecond.

**ليه؟** الـ notifier chain في الـ Linux بتشغّل الـ callbacks بالترتيب. لما الـ CCF يقرر يغير تردد `cpu_clk`، بتحصل الأحداث دي:

1. الـ CCF بيطلق `PRE_RATE_CHANGE` على `cpu_clk`
2. `g12a_cpu_clk_mux_nb` بياخد الـ notification وبيعمل delay
3. **بعدين** الـ notifiers التانية (postmux أو sys_pll) بتتفعل
4. الـ CCF يغير الـ rate
5. الـ CCF بيطلق `POST_RATE_CHANGE`
6. تاني delay

الـ delay ده بيدي وقت للـ hardware إنه يستقر بعد أي تغيير في الـ clock path. في الـ SoCs زي G12A، الـ hardware محتاج وقت بعد ما تغير الـ mux عشان الـ clock signal يستقر.

**`NOTIFY_OK`** يعني: أنا عملت اللي عليّ، تقدر تكمل.
**`NOTIFY_DONE`** يعني: مش من شغلي، مش مهتم.

#### 2. `g12a_cpu_clk_postmux_notifier_cb` — قلب الـ fix

```c
static int g12a_cpu_clk_postmux_notifier_cb(struct notifier_block *nb,
                        unsigned long event, void *data)
{
    struct g12a_cpu_clk_postmux_nb_data *nb_data =
        container_of(nb, struct g12a_cpu_clk_postmux_nb_data, nb);
```

أول حاجة: `container_of` يجيب الـ struct الكاملة من الـ `nb` pointer. ده لأن الـ kernel بيبعت الـ `nb` pointer بس للـ callback، فمحتاجين نوصل للمؤشرات الـ `xtal`، `cpu_clk_dyn`، إلخ.

##### الـ PRE_RATE_CHANGE case:

```c
case PRE_RATE_CHANGE:
    /*
     * This notifier means cpu_clk_postmux0 clock will be changed
     * to feed cpu_clk, this is the current path :
     * cpu_clk
     *    \- cpu_clk_dyn
     *          \- cpu_clk_postmux0
     *                \- cpu_clk_muxX_div
     *                      \- cpu_clk_premux0
     *                            \- fclk_div3 or fclk_div2
     *             OR
     *                \- cpu_clk_premux0
     *                      \- fclk_div3 or fclk_div2
     */

    /* Setup cpu_clk_premux1 to xtal */
    clk_hw_set_parent(nb_data->cpu_clk_premux1,
                      nb_data->xtal);

    /* Setup cpu_clk_postmux1 to bypass divider */
    clk_hw_set_parent(nb_data->cpu_clk_postmux1,
                      nb_data->cpu_clk_premux1);

    /* Switch to parking clk on cpu_clk_postmux1 */
    clk_hw_set_parent(nb_data->cpu_clk_dyn,
                      nb_data->cpu_clk_postmux1);

    /*
     * Now, cpu_clk is 24MHz in the current path :
     * cpu_clk
     *    \- cpu_clk_dyn
     *          \- cpu_clk_postmux1
     *                \- cpu_clk_premux1
     *                      \- xtal
     */

    udelay(100);

    return NOTIFY_OK;
```

**الخطوات:**

1. **`clk_hw_set_parent(premux1, xtal)`** — نخلي الـ premux1 يشاور على الـ xtal (24MHz). ده آمن لأن premux1 لسه مش active.
2. **`clk_hw_set_parent(postmux1, premux1)`** — نخلي الـ postmux1 يـ bypass الـ divider ويشاور مباشرة على premux1. كمان آمن.
3. **`clk_hw_set_parent(cpu_clk_dyn, postmux1)`** — **هنا بيحصل الانتقال الفعلي**: الـ CPU بقى شغال على 24MHz (xtal). الـ postmux0 والـ postmux1 بيشاوروا على نفس الـ xtal بس، فالـ glitch مش موجود — لأن الـ CPU لسه شغال بنفس الـ signal source قبل وبعد.

**بعدين** `udelay(100)` عشان الـ hardware يستقر على الـ مسار الجديد.

الـ CCF دلوقتي قادر يغير الـ premux0 وكل الـ tree بتاعته بدون ما يأثر على الـ CPU لأن الـ CPU مش شغال عليها.

##### الـ POST_RATE_CHANGE case:

```c
case POST_RATE_CHANGE:
    /*
     * The cpu_clk_postmux0 has been updated, now switch back
     * cpu_clk_dyn to cpu_clk_postmux0 and take the changes
     * in account.
     */

    /* Configure cpu_clk_dyn back to cpu_clk_postmux0 */
    clk_hw_set_parent(nb_data->cpu_clk_dyn,
                      nb_data->cpu_clk_postmux0);

    /*
     * new path :
     * cpu_clk
     *    \- cpu_clk_dyn
     *          \- cpu_clk_postmux0
     *                \- cpu_clk_muxX_div
     *                      \- cpu_clk_premux0
     *                            \- fclk_div3 or fclk_div2
     *             OR
     *                \- cpu_clk_premux0
     *                      \- fclk_div3 or fclk_div2
     */

    udelay(100);

    return NOTIFY_OK;
```

بعد ما الـ CCF خلّص تغيير الـ premux0 tree، بنعمل خطوة واحدة: نحوّل الـ `cpu_clk_dyn` يرجع يشاور على `cpu_clk_postmux0` اللي بقى بالتردد الجديد.

الـ `udelay(100)` بعدها عشان الـ hardware يستقر قبل ما الـ kernel يعتبر إن الـ transition خلص.

#### 3. `g12a_sys_pll_notifier_cb` — التعامل مع تغيير الـ PLL نفسه

```c
static int g12a_sys_pll_notifier_cb(struct notifier_block *nb,
                    unsigned long event, void *data)
{
    struct g12a_sys_pll_nb_data *nb_data =
        container_of(nb, struct g12a_sys_pll_nb_data, nb);
```

##### الـ PRE_RATE_CHANGE case:

```c
case PRE_RATE_CHANGE:
    /*
     * This notifier means sys_pll clock will be changed
     * to feed cpu_clk, this the current path :
     * cpu_clk
     *    \- sys_pll
     *          \- sys_pll_dco
     */

    /* Configure cpu_clk to use cpu_clk_dyn */
    clk_hw_set_parent(nb_data->cpu_clk,
                      nb_data->cpu_clk_dyn);

    /*
     * Now, cpu_clk uses the dyn path
     * cpu_clk
     *    \- cpu_clk_dyn
     *          \- cpu_clk_dynX
     *                \- cpu_clk_dynX_sel
     *                      \- cpu_clk_dynX_div
     *                            \- xtal/fclk_div2/fclk_div3
     *                   \- xtal/fclk_div2/fclk_div3
     */

    udelay(100);

    return NOTIFY_OK;
```

لما الـ CCF يحب يغير تردد الـ sys_pll (اللي بيشغّل الـ CPU على تردد جيجاهرتز)، محتاج يعمل reprogram للـ PLL، وده بيستلزم إن الـ PLL يعدي بفترة unstable.

الحل: قبل ما الـ PLL يتغير، بنحوّل الـ `cpu_clk` عشان يشاور على `cpu_clk_dyn` (اللي بيشغّل على fclk_div أو xtal). الـ CPU بيظل شغال، بس على تردد أقل وآمن.

##### الـ POST_RATE_CHANGE case:

```c
case POST_RATE_CHANGE:
    /* Configure cpu_clk to use sys_pll */
    clk_hw_set_parent(nb_data->cpu_clk,
                      nb_data->sys_pll);

    udelay(100);

    /* new path :
     * cpu_clk
     *    \- sys_pll
     *          \- sys_pll_dco
     */

    return NOTIFY_OK;
```

بعد ما الـ PLL استقر على الـ تردد الجديد، بنرجع نحوّل الـ `cpu_clk` إنه يشاور على `sys_pll`. الـ CPU دلوقتي شغال على التردد الجديد.

#### 4. `meson_clk_cpu_dyndiv_set_rate` — الكتابة الآمنة للـ divider

```c
static int meson_clk_cpu_dyndiv_set_rate(struct clk_hw *hw, unsigned long rate,
                                          unsigned long parent_rate)
{
    struct clk_regmap *clk = to_clk_regmap(hw);
    struct meson_clk_cpu_dyndiv_data *data = meson_clk_cpu_dyndiv_data(clk);
    unsigned int val;
    int ret;

    ret = divider_get_val(rate, parent_rate, NULL, data->div.width, 0);
    if (ret < 0)
        return ret;

    val = (unsigned int)ret << data->div.shift;

    /* Write the SYS_CPU_DYN_ENABLE bit before changing the divider */
    meson_parm_write(clk->map, &data->dyn, 1);

    /* Update the divider while removing the SYS_CPU_DYN_ENABLE bit */
    return regmap_update_bits(clk->map, data->div.reg_off,
                              SETPMASK(data->div.width, data->div.shift) |
                              SETPMASK(data->dyn.width, data->dyn.shift),
                              val);
};
```

**الخطوات:**

1. **`divider_get_val`** — بيحسب قيمة الـ divider اللي محتاجينها من الـ rate المطلوبة والـ parent rate.
2. **`meson_parm_write(dyn, 1)`** — بيكتب `1` في الـ `SYS_CPU_DYN_ENABLE` bit. ده بيقول للـ hardware: "جهّز نفسك لـ transition".
3. **`regmap_update_bits`** — في نفس الـ atomic write، بيكتب قيمة الـ divider الجديدة **ويمسح** الـ `dyn` bit. الـ hardware بيشوف إن الـ dyn bit كان `1` وبقى `0` في نفس اللحظة اللي اتغير فيها الـ divider، وده بيـ trigger الـ hardware-level glitch-free transition.

ده تصميم hardware-aware — الـ hardware نفسه بيعمل الـ transition بشكل آمن لما شاف الـ dyn bit sequence.

#### 5. `meson_g12a_dvfs_setup_common` — الـ setup المشترك بين G12A وG12B

```c

**أول خطوة**: بيجيب الـ `xtal` clock handle من الـ clock tree. بيستخدم `clk_hw_get_parent_by_index` على `CLKID_CPU_CLK_DYN1_SEL` (اللي هو `cpu_clk_premux1`) ويجيب الـ parent رقم 0 اللي هو الـ xtal.

```

الـ function دي بتتشارك فيها G12A وG12B لأن الكلسترات الأولى فيهم نفس البنية (`cpu_clk_dyn` + `cpu_clk_premux0`).

#### 8. `meson_g12a_probe` — الـ probe الجديد

```c
static int meson_g12a_probe(struct platform_device *pdev)
{
    const struct meson_eeclkc_data *eeclkc_data;
    const struct meson_g12a_data *g12a_data;
    int ret;

    eeclkc_data = of_device_get_match_data(&pdev->dev);
    if (!eeclkc_data)
        return -EINVAL;

    ret = meson_eeclkc_probe(pdev);
    if (ret)
        return ret;

    g12a_data = container_of(eeclkc_data, struct meson_g12a_data,
                             eeclkc_data);

    if (g12a_data->dvfs_setup)
        return g12a_data->dvfs_setup(pdev);

    return 0;
}
```

**الخطوات:**

1. `of_device_get_match_data` — بتجيب الـ data المخصصة من الـ `of_device_id` table. بترجع `const struct meson_eeclkc_data *` (عشان الـ table محطوط فيها `&g12a_clkc_data.eeclkc_data`).
2. `meson_eeclkc_probe` — بيعمل الـ initialization العادي للـ clock controller (يسجل الـ clocks في الـ CCF).
3. `container_of` — بيرجع من الـ `eeclkc_data` pointer للـ `meson_g12a_data` الكاملة.
4. لو في `dvfs_setup` function، بيستدعيها. ده بيسمح بإن كل SoC يعمل الـ DVFS setup الخاص بيه.

### الـ Patterns

#### `container_of` Pattern

الـ `container_of` هو macro بيحوّل pointer على member جوه struct لـ pointer على الـ struct نفسها.

```c
#define container_of(ptr, type, member) \
    ((type *)((char *)(ptr) - offsetof(type, member)))
```

**مثال من الكود:**

```c
struct g12a_cpu_clk_postmux_nb_data *nb_data =
    container_of(nb, struct g12a_cpu_clk_postmux_nb_data, nb);
```

الـ kernel بيبعت للـ callback pointer على `nb` بس. عن طريق `container_of`، بنطرح الـ offset بتاع `nb` من عنوانه، فنوصل للبداية الحقيقية للـ `g12a_cpu_clk_postmux_nb_data`. ده بيسمح إننا نـ embed أي data إضافية مع الـ `notifier_block` بدون ما نغير في الـ kernel API.

---

#### Notifier Chain Pattern

**الـ** notifier chain في Linux هي آلية publish-subscribe بين أجزاء الـ kernel. الـ CCF بيستخدمها عشان يعلم الـ drivers قبل وبعد تغيير الـ clock rate.

**التسجيل:**

```c
int clk_notifier_register(struct clk *clk, struct notifier_block *nb);
```

**الـ events:**
- `PRE_RATE_CHANGE` — قبل ما الـ CCF يغير الـ rate
- `POST_RATE_CHANGE` — بعد ما الـ CCF غيّر الـ rate
- `ABORT_RATE_CHANGE` — لو التغيير فشل

**الـ lifecycle:**
1. الـ driver بيسجّل الـ `notifier_block` على clock معين.
2. لما الـ CCF يحب يغير rate الـ clock ده، بيعمل `srcu_notifier_call_chain` وبيستدعي كل الـ callbacks المسجلة.
3. الـ callback بيتعامل مع الـ event وبيرجع `NOTIFY_OK` أو `NOTIFY_DONE`.

---

#### Clock Mux Reparenting Pattern

**الـ** `clk_hw_set_parent` بتغير الـ parent بتاع مسار معين في الـ clock tree:

```c
clk_hw_set_parent(nb_data->cpu_clk_dyn, nb_data->cpu_clk_postmux1);
```

ده بيكتب في الـ mux register الـ selector value المقابل للـ parent الجديد. الـ CCF بيعرف الـ hardware mapping من الـ `clk_regmap_mux_data`.

**ليه** `CLK_SET_RATE_NO_REPARENT` مهم للـ parking subtree؟ لأن الـ CCF لما بيحسب أحسن rate، ممكن يقرر يغير الـ parent لو لاقى rate أفضل عند parent تاني. إننا نحط `CLK_SET_RATE_NO_REPARENT` على الـ parking subtree (premux1, postmux1) بيمنع الـ CCF من إنه يـ mess بيهم أثناء الـ DVFS — ده الـ subtree اللي إحنا بس اللي بنتحكم فيه من الـ notifier.

---

#### Dynamic Divider Hardware Protocol

```c
/* Write the SYS_CPU_DYN_ENABLE bit before changing the divider */
meson_parm_write(clk->map, &data->dyn, 1);

/* Update the divider while removing the SYS_CPU_DYN_ENABLE bit */
return regmap_update_bits(clk->map, data->div.reg_off,
                          SETPMASK(data->div.width, data->div.shift) |
                          SETPMASK(data->dyn.width, data->dyn.shift),
                          val);
```

الـ Amlogic hardware بيوفر protocol خاص للـ CPU clock divider. لما `SYS_CPU_DYN_ENABLE` يتكتب `1` ثم يتمسح مع كتابة الـ divider الجديد في نفس الـ transaction — الـ hardware بيعمل الـ transition بدون glitch باستخدام internal latch/shadow register. ده hardware-specific protocol ومش موثق في datasheet عام.

**ترتيب العمليات في الـ probe:**

```
meson_g12a_probe()
    │
    ├─ of_device_get_match_data()
    │    └─ بترجع &g12a_clkc_data.eeclkc_data
    │
    ├─ meson_eeclkc_probe()   ← الـ clock registration الأصلي
    │    ├─ regmap setup
    │    ├─ clock registration في CCF
    │    └─ hw_onecell_data setup
    │
    ├─ container_of(eeclkc_data, struct meson_g12a_data, eeclkc_data)
    │    └─ بيلاقي مؤشر على g12a_clkc_data كاملاً
    │
    └─ g12a_data->dvfs_setup(pdev)   ← meson_g12a_dvfs_setup()
         ├─ meson_g12a_dvfs_setup_common()
         │    ├─ clk_hw_get_parent_by_index() — بيجيب xtal
         │    ├─ clk_notifier_register(cpu_clk_postmux0, postmux_nb)
         │    └─ clk_notifier_register(cpu_clk_dyn, mux_nb)
         │
         ├─ clk_notifier_register(cpu_clk, mux_nb)
         └─ clk_notifier_register(sys_pll, sys_pll_nb)
```

**ليه الترتيب مهم؟**

الـ `meson_eeclkc_probe` لازم يخلص أول عشان الـ clocks تتسجل في الـ CCF وتبقى قابلة للـ lookup بالاسم. لو عكسنا الترتيب، الـ `__clk_lookup` هيفشل لأن الـ clocks لسه مش مسجلة.

**ماذا لو فشل التسجيل؟**

لو أي `clk_notifier_register` رجعت error، الـ function بترجع الـ error فوراً (`return ret`). ده بيخلي الـ `probe` يفشل، والـ kernel بيـ unload الـ driver. المشكلة: الـ notifiers اللي اتسجلوا بالفعل **مش بتتـ unregister** — ده error path leak. ده بيُذكر في الـ FIXME أو كـ limitation مضمنة في الكود.

### الـ FIXME — الحل المؤقت

الـ diff نفسه ما فيهوش FIXME صريح، لكن في جزء من الكود فيه حل مؤقت واضح:

```c
notifier_clk_name = clk_hw_get_name(&g12a_cpu_clk_postmux0.hw);
notifier_clk = __clk_lookup(notifier_clk_name);
ret = clk_notifier_register(notifier_clk, &g12a_cpu_clk_postmux0_nb_data.nb);
```

**المشكلة الضمنية**: `__clk_lookup` هي internal function (prefix underscore) وبتـ return `struct clk *` اللي هي الـ legacy handle. الـ API "الصح" في الـ CCF الحديث كانت هتستخدم `clk_hw_register_notifier` أو نظير مباشر للـ `clk_hw`، لكن ده API ما كانش موجوداً وقتها.

**الحل المؤقت الآخر**: الـ notifiers بتتسجل بـ order غير مضمون — لو حصل تسابق بين notifiers، الـ delay الـ `udelay(100)` هي اللي بتضمن التزامن بدل آلية أصرح. المؤلف نفسه في commit message بيقول:

> "A third small notifier is added on each cpu_clk / cpub_clk and cpu_clk_dyn, add a small delay at PRE_RATE_CHANGE/POST_RATE_CHANGE to let the other notifiers change propagate before changing the cpu_clk_premux0 and sys_pll clock trees."

الـ `udelay(100)` هنا بيشتغل كـ **implicit synchronization** بدل explicit ordering. الحل الأفضل كان يستخدم explicit notifier priority (الـ `notifier_block.priority` field) عشان يضمن تنفيذ الـ notifiers بترتيب صح.

**ليه مش عاملين unregister للـ notifiers؟** مافيش `remove` callback في الـ driver، يعني الـ notifiers بتظل مسجلة طول فترة حياة الـ kernel. ده acceptable لأن الـ clock controller ما بيتـ unload، لكنه technically leak في case الـ probe failure.
